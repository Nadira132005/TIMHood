You are a senior product architect helping me turn a product specification into a clean technical architecture overview for a hackathon app.

Your task is to analyze the @INIT.md sections and, for each section, produce an architectural breakdown that is detailed enough to guide backend/frontend planning, but does NOT go into implementation-specific code, framework choices, SQL syntax, or endpoint-level details.

Important constraints:

- Focus on architecture, domain modeling, system boundaries, object relationships, permissions, states, and interaction flows
- Do NOT write implementation details such as exact database migrations, API routes, controller logic, UI code, libraries, or deployment setup
- Keep the output practical and structured for a small hackathon team
- Prefer simplicity and MVP realism over enterprise complexity
- When something is too complex for MVP, explicitly label it as “post-MVP” or “optional”
- Preserve the product boundaries exactly:
  - Posts = sharing content
  - Events = scheduled activities with moderated participation
  - Pings = urgent community help requests
  - Services = non-urgent structured help requests with applicant selection

For each major section in the INIT document, produce the following:

1. Purpose

- What this section is responsible for in the product
- Why it exists
- What user problem it solves

2. Architectural Role

- Whether this is a core domain, supporting domain, or cross-cutting capability
- Which other sections it depends on
- Which other sections depend on it

3. Main Domain Entities

- List the main conceptual entities involved
- Briefly describe each entity in product/architecture terms
- Include only meaningful domain objects, not low-level implementation artifacts

4. State and Lifecycle

- Describe the important states for the entities in this section
- Explain how objects move between states
- Include moderation/approval states where relevant

5. Permissions and Actors

- Identify the actor types involved (e.g. member, admin, verified user, requester, applicant)
- Describe what each actor is allowed to do in this section
- Clearly note approval responsibilities where relevant

6. Relationships and Boundaries

- Explain how this section connects to the other sections
- Clarify ownership rules
- Clarify which objects belong to a community, which are user-owned, and which are cross-community
- Explicitly call out boundaries that prevent feature overlap or ambiguity

7. Core User Flows

- Describe the main flows a user would experience
- Use short structured flows, such as:
  - create
  - discover
  - request/apply/respond
  - approve/reject
  - complete/expire
- Focus on behavior, not UI implementation

8. Architecture Notes

- Mention any important architectural concerns:
  - moderation points
  - trust boundaries
  - visibility rules
  - privacy concerns
  - discoverability
  - cross-community sharing
  - expiration logic
- Keep these at the system-design level, not implementation level

9. MVP Scope Recommendation

- Say what should be included in a one-day hackathon MVP
- Say what should be postponed
- Be opinionated and practical

10. Candidate Data Model Shape

- Provide a lightweight conceptual model only
- Example format:
  - Community
  - Event
  - EventShare
  - EventParticipationRequest
- For each one, include just a short description of what it represents
- Do NOT define full schemas, field types, SQL, or implementation-level attributes unless absolutely necessary for architecture clarity

Output format requirements:

- Organize the answer section by section, following the INIT document structure
- Use clear headings
- Be explicit and unambiguous
- Where there is a product rule, call it out as a “Rule”
- Where there is a potential ambiguity, call it out as a “Boundary”
- Where something is risky or too big for MVP, call it out as “Hackathon caution”

Additional product rules you must preserve:

Events:

- An event has exactly one origin community
- It may be shared into other communities with admin approval
- Users do not directly join; they request participation
- Participation must be accepted by admins
- Events are moderated, not first-come first-served

Pings:

- A community decides whether pings are enabled
- Only verified users can create pings
- Pings are urgent and short-lived
- Pings support a lightweight “WILL HELP” response and visible response count
- Pings should remain clearly distinct from services

Services:

- A service belongs to one community
- Services are also discoverable in a dedicated Services section
- Services are non-urgent
- Users apply to help
- The creator selects one helper in MVP
- Services should not drift into becoming events or pings

Communities:

- Communities are the main trust and moderation layer
- They can configure entry, posting, replies, pings, and optional features such as daily riddles

Profiles:

- Profiles power lightweight discovery and social matching
- Keep profile-based matching soft and non-invasive in architecture terms

Important writing style:

- Write like an architect producing a technical overview for product and engineering alignment
- Be concise but not shallow
- Avoid implementation detail
- Avoid generic filler
- Prefer crisp product architecture language

At the end, produce:

1. A short “System-wide Architecture Summary”
2. A short “MVP Cut Recommendation”
3. A short “Ambiguities That Still Need Product Decisions” section
